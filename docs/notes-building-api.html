A. 模型與 API 基礎

Claude 三模型選型D4
OpusSonnetHaiku
定位最高智能、深度推理/規劃智能/速度/成本平衡，強 coding最快最省，無推理
適合複雜多步、需 reasoning多數實務、精準改碼即時互動、高量處理
選型：要聰明→Opus，要快→Haiku，平衡→Sonnet。實務常在同一應用依任務混用多模型。三者共通(文字/coding/看圖)，差在最佳化取向。
API 存取流程（5 步）
  • Client 把使用者輸入送到你的 server（絕不從 client 直接打 Anthropic API → 保護 key）。
  • Server 用 SDK(Python/TS/JS/Go/Ruby)或 HTTP 打 API，必帶：api key + model + messages + max_tokens。
  • 文字生成 4 階段：Tokenization(切 token)→Embedding(轉數字)→Contextualization(依鄰近 token 調整語意)→Generation(算下一字機率+隨機挑，重複)。
  • 停止：達 max_tokens 或生成 end_of_sequence token。
  • 回傳：text + usage + stop_reason → server → client。
發 RequestD4
  • client.messages.create(model, max_tokens, messages)。max_tokens=安全上限，非目標長度。
  • 訊息 = list of dict：{"role":"user"/"assistant", "content":"..."}。
  • 取純文字：message.content[0].text。
  • setup：pip install anthropic python-dotenv → .env 放 ANTHROPIC_API_KEY(版控忽略)→ dotenv 載入 → 建 client。
多輪對話D5
⚠️ API 完全 stateless，不存任何訊息，每次請求獨立、無記憶。
解法：自己在程式維護 messages list，每次把完整對話史一起送。流程：送 user → 收 assistant → append 進 history → 加新 user → 再送完整 history。helper：add_user_message / add_assistant_message / chat。

B. 控制輸出

System PromptsD4 用 system= 傳字串。控制「怎麼回」不是「回什麼」：第一行給角色(You are a patient math tutor)+行為指令。同問題因角色不同而不同對待。None 時整個排除 system 參數。
Temperature（0–1）D4 控隨機性。0=deterministic(永選最高機率 token)；高=更創意/意外。低→擷取/事實(一致)；高→腦力激盪/創作。高值只是提高變異機率，不保證不同。
Response Streaming chunk-by-chunk 顯示(回應可能 10–30 秒)。事件：message_start→content_block_start→content_block_delta(含文字 chunk，最重要)→content_block_stop/message_stop。client.messages.stream() 的 text_stream 取純文字；get_final_message() 組完整訊息存 DB。
Pre-fill Assistant + Stop SequencesD4
  • Pre-fill：在 messages 尾端手動加一則 assistant 訊息 → Claude 從那段結尾接續(要自己把 prefill+回應拼起來)。
  • Stop sequence：生成到指定字串立即停，該字串不含在輸出。
Structured Data（prefill + stop）D4 要 raw JSON/code 無多餘說明：user 要資料 → assistant prefill ```json → stop ```。任何結構化型別通用，輸出可直接 parse。
Extended ThinkingD5 生成前先推理，提升複雜任務準度但加成本/延遲。thinking_budget 最少 1024，max_tokens 必須 > thinking_budget。回應含 thinking block(帶簽章防竄改)+ text block。prompt 優化後仍不夠準才開，用 eval 判斷。

C. Prompt Engineering & Eval

Prompt EvaluationD4 工程師常測不足：測一兩次就上線(陷阱)/自訂幾例微調(陷阱)/跑 eval pipeline 客觀評分(正解)。
Eval Workflow（6 步）D4 ①寫 prompt 草稿 ②建 eval dataset(3 例到數千，手寫或 LLM 產)③把 input 帶進 prompt ④收 Claude 回應 ⑤grader 評分(如 1–10)取平均 ⑥依分數迭代、比版本。無標準方法，可自建簡單版。
Grading 三種D4
  • Code grader：程式檢查(長度/關鍵字/語法 valid/可讀性)。validate_json/python/regex valid=10、error=0。
  • Model grader：另發一次 API 評分，要 prompt 求strengths/weaknesses/reasoning/score(別只要 score 否則給中庸分)，JSON 輸出。
  • Human grader：最彈性但耗時。
最終分常 = (model_score + syntax_score)/2。
Prompt 技巧D4
  • Clear & Direct：第一行用 action verb 開頭直指任務(Write/Identify/Generate…)。
  • Being Specific：Type A attributes(要的品質/格式，幾乎每個 prompt 都用)+ Type B steps(給推理步驟，複雜題用)。範例分數 3.92→7.86。
  • XML tags：把插入的內容用描述性標籤包(<sales_records> 勝 <data>)，幫模型分辨區塊；短內容也值得包。
  • Examples(one/multi-shot)：用 XML 包 sample input+ideal output，放在指令之後；加「為何這是理想輸出」的說明；corner case(諷刺偵測)特別有效；用 eval 最高分例當範本。

D. Tool Use

概念與流程D2 讓 Claude 取訓練資料外的即時資訊。5 步：送 request+工具 → Claude 判斷要不要+要哪個 → server 跑 code 取資料 → 帶資料再送 → Claude 產最終回應。
Tool Functions & SchemasD2
  • Tool function=普通 Python 函式；命名要描述性、驗證輸入、error 訊息要有意義(error 給 Claude 看，它可重試)。
  • Schema：name + description(3–4 句：做什麼/何時用/回什麼)+ input_schema(JSON schema)。用 ToolParam() 包避免型別錯。產 schema 技巧：拿函式到 claude.ai + 附官方 tool use 文件，請它生。
Message Blocks & Tool ResultsD2
  • assistant 回應可含多 block：text block + tool_use block(函式名+參數)。
  • tool result block：tool_use_id(配對原請求)+ content(輸出轉字串)+ is_error。放在 user 訊息，不是 assistant。
  • follow-up 要帶完整 history + 原本的 tool schemas(即使不再用工具)。
  • 要把整個 response.content(所有 block) append 進 history，不是只 text。
Multi-turn Tool（while loop）D1 一個問題可能鏈式用多工具。run_conversation：呼叫 Claude → 加回應進 history → 檢查 stop_reason：!="tool_use" 就 break；="tool_use" 就 run_tools(過濾 type=tool_use 的 block、各跑、回 tool_result list)→ 加結果為 user 訊息 → 重複。run_tool=用 if 分派工具名。try/except 包執行(成功 is_error=false，失敗 true+error)。
Batch / Fine-grained / StructuredD2
  • Batch tool：Claude 少主動並行 → 給一個 batch 工具收 invocations list，一次跑多工具(單一 request-response)。
  • Fine-grained tool calling(fine_grained:true)：關 API 端 JSON 驗證，chunk 即時送(較快但可能 invalid JSON，要 client 端處理)。預設=慢但驗證過。
  • Tools for structured data：比 prefill 法更可靠但較複雜。定 JSON schema 當工具輸入，強制 tool_choice={"type":"tool","name":"..."}，從 response.content[0].input 取結構化資料(不需 tool result)。
內建工具D2 Text Editor(唯一內建進 Claude 的 schema，實作要自己寫檔案操作；schema type 依模型版本不同)；Web Search(web_search_20250305，max_uses 預設 5、allowed_domains 限域，回 search result + citation block，無需自寫 code)。

E. RAG（Retrieval Augmented Generation）

概念D3 大文件(百千頁)查特定資訊。全塞 prompt=撞 token 上限/變貴變慢/效果降。RAG：切 chunk → 只找相關 chunk 放進 prompt。省成本、可擴展，但要前處理+搜尋機制。
Chunking 策略D3 chunking 品質直接決定 RAG 成敗。①Size-based(等長，最常用，會切斷字→用 overlap 補上下文)②Structure-based(依標題/段落，需結構化文件如 markdown)③Semantic-based(NLP 依語意分組，最進階)。無萬用最佳法，看文件結構與用途。
Embeddings & Semantic Search embedding=文字語意的數字向量(-1~+1)。semantic search 用向量找相關 chunk(語意相似而非關鍵字)。Anthropic 建議 Voyage AI 產 embedding。
完整 RAG Flow（7 步） chunk → embed → normalize(scale 到 1.0)→ 存 vector DB → query 轉 embedding → cosine similarity 找最近 → 組 prompt。cosine sim 越近 1 越像；cosine distance=1−sim 越近 0 越像。前處理(1–4)+ 即時檢索(5–7)。
進階檢索
  • BM25(lexical)：補語意搜尋抓不到的精確字。稀有字權重高、常見字低。Hybrid=語意+BM25 並行後合併。
  • Multi-index + RRF(Reciprocal Rank Fusion)：RRF=Σ 1/(rank+1)，兩法都排前的文件勝出。
  • Reranking：檢索後用 LLM 依相關性重排(用 doc id 省 token)，增準度但加延遲。
  • Contextual Retrieval：embed 前先給每個 chunk 加「它在原文的情境」→ chunk 不失上下文。大文件用選擇性情境(開頭 1–3 chunk + 目標前幾個)。

F. 進階功能

Image / PDF image block(base64 或 URL，單訊息可多張，最多 100，耗 token)。圖片準度靠 prompt 功力(step-by-step/多 shot/驗證步驟)，非圖質。PDF：type=document、media=application/pdf，可讀文字+圖+表+圖表。
Citations "citations":{"enabled":true} + title。PDF 用 citation_page_location、純文字用 citation_char_location。讓使用者查證來源，非憑記憶回答。
Prompt CachingD5 重用前次輸入處理的計算，加速+省錢。
規則：快取 1 小時；要手動加 cache_control:{"type":"ephemeral"}(需長格式 text block)；順序 tools → system → messages；breakpoint 前所有內容一起快取；前面任何改動使整個快取失效；最少 1024 token 才快取；最多 4 個 breakpoint(多層、可部分命中)。
usage：cache_creation_input_tokens(首次寫)/cache_read_input_tokens(命中讀)。最佳用途=重複的 system prompt / tool 定義 / 靜態前綴。
Code Execution & Files API Files API：先上傳拿 file ID，之後用 ID 引用(不用每次塞原始資料)。Code Execution：Claude 在隔離 Docker跑 Python(無網路，資料靠 Files API 進出)，可多次執行、產檔(圖/報告)供下載。

G. MCP（Model Context Protocol）

概念D2 通訊層，把 tool 的定義與執行從你的 app 搬到 MCP server，省去自己寫一堆 tool schema/函式(如 GitHub 要做 repo/PR/issue 一堆工具)。MCP 與 tool use 互補：MCP 決定「誰做」(server vs 你)，都還是 tools。server 常由服務商官方出。
Client & 訊息D2 MCP client=你的 server 與 MCP server 的介面，transport 無關(stdio/HTTP/WebSocket)。四訊息：list tools request/result、call tool request/result。流程：user→server 向 client 要工具清單→轉發 MCP server→回清單→送 Claude→Claude 要執行→client 呼叫 MCP server 跑→結果回鏈。
SDK 定義D2 @mcp.tool 從 Python 函式簽章自動生 schema(免手寫 JSON)；參數用 Field() 加描述。debug 用 MCP Inspector(mcp dev server.py 開瀏覽器手動測工具)。
Resources vs PromptsD2
  • Resources：server 曝資料給 client 讀。direct(靜態 URI docs://documents)/templated(docs://documents/{id})。@mcp.resource + MIME type。Resource 主動提供資料(@ 提及即抓)，Tool 被動執行動作(Claude 決定才叫)。
  • Prompts：server 端預先寫好、測過的高品質 prompt 模板，client 端變 slash command 選用。@mcp.prompt 回 messages list。

H. Claude Code & Agents

Claude Code 基礎D3
  • 裝：Node.js → npm install → claude 登入。內建搜尋/讀/改檔 + web fetch/terminal + MCP client。
  • init=掃 codebase 架構/風格產 claude.md(後續自動載入的 context)。memory 三種：Project(共享)/Local/User。# 加筆記進 memory；可手動改 claude.md 或重跑 init。
  • Prompt 法：三步(找檔分析→描述功能請它規劃不寫碼→請它實作)或 TDD(給 context→請它建議測試→選並實作→寫碼到測試過)。「更詳細指令=更好結果」，當協作工程師不是產碼機。
Claude Code 進階D3
  • 加 MCP：claude mcp add [name] [startup-cmd]，重啟後可用(Sentry/Jira/Slack…)。
  • 平行化：多 Claude 同改同檔會衝突 → 用 git worktrees 隔離工作區(各對應一 branch)。自訂命令放 .claude/commands/*.md(用 $ARGUMENTS)。
  • 自動除錯：GitHub Action 每日抓 CloudWatch log → Claude 找錯/去重/產修 → 開 PR(抓 production-only 錯，如環境間 config 差異)。
Computer Use Claude 靠視覺觀察+控制(截圖→點/打字/導航)做多步任務(QA 測試自動化)。在隔離 Docker 跑。本質=tool 系統(schema 有 mouse move/click/screenshot)+ 開發者提供的運算環境(Anthropic 給 reference Docker)。Claude 不直接操控電腦。
Agents vs WorkflowsD1
判準：步驟清楚→workflow；細節不明→agent。
  • Workflow=為特定問題、步驟預定的一連串 Claude 呼叫。模式：Chaining(拆序列步驟，破解 constraint-heavy prompt)、Parallelization(拆平行子任務→aggregator 合)、Routing(先分類→路由到專門 pipeline)、Evaluator-Optimizer(producer 產→evaluator 評→loop 到接受，如 圖→3D 模型)。
  • Agent=自己擬計畫、用工具完成任務(細節不明時用)。
關鍵：認出 pattern ≠ 自動實作，仍要寫 code。
Agent 工具設計原則D1D2 給 agent 通用/抽象工具(bash、web_fetch、file_write)，不是超專用(refactor_tool、install_dependencies)。小組彈性工具創意組合 → 動態解題+意外用途。範例：get_datetime+add_duration+set_reminder 可組出各種時間任務。
Environment InspectionD1 agent 每個動作後要檢查環境+動作結果才知進度/抓錯(無法預測按鈕點下去的結果)。computer use 每動作後截圖；改檔前先讀現況。讓 agent 不盲目操作、能適應意外。
Workflow vs Agent（取捨）D1
步驟已知→workflow；步驟未知→agent。
workflow：拆小子任務、易測(執行序已知)、成功率高、需特定輸入。agent：創意應變、難測(路徑不可測)、成功率較低、自產輸入可要更多資訊。建議：優先 workflow 求可靠，真的需要彈性才用 agent。先可靠解決問題，創新其次(用戶要 100% 能動的產品，不是花俏 agent)。
