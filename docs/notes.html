① Introduction to Subagents

Academy 課程 · 對應 CCAR-F Domain 1（Agentic Architecture 27%）+ Domain 3（Claude Code Config）
🎬 影片 1：概念 · 影片 2：建立 · 影片 3：有效 pattern · 影片 4：何時用/別用 · 課程頁

1. 什麼是 subagent

  • Claude Code 可委派任務的專用助手：各自跑在獨立 context window，做完只把摘要回主線。
  • 中間所有動作（file read、search、tool call）都留在自己的 context，不會塞爆主對話。

2. 為什麼重要（核心 = context 隔離）

  • 每次對話都在累積主 context window，工具呼叫/讀檔/搜尋結果都存那；容量有限，填滿後 Claude 開始失去對前面內容的追蹤。
  • subagent 開一個獨立 context 解決此問題。它收兩樣東西：① config 檔的 custom system prompt（定義角色/行為）② parent 依你需求寫的 task description。
  • 然後獨立工作，完成後只回摘要，整段 subagent 對話被丟棄。
  • Tradeoff：主 context 乾淨，但你失去「它如何得到結論」的可見性。

3. 實例

問「哪個 service 處理退款」：
· 無 subagent → Claude 可能讀 15 個檔、跑多次搜尋、追多個函式，全塞進你的 context（你其實只要一個事實）。
· 有 subagent → Explore subagent 在自己 context 挖，回一個聚焦答案。你的主 context 只記「問題 + 摘要」，不記那 15 個檔。

4. 內建 subagents

Subagent用途
General purpose需要「探索 + 行動」的多步驟任務
Explore快速搜尋/導航 codebase
Planplan mode 時，先研究分析 codebase 再提計畫

5. 自訂 subagent

本質：markdown 檔 + YAML frontmatter，告訴 Claude「何時用」與「如何 behave」。

建立流程（/agents slash command → Create new agent）：
  • Scope：Project-level（僅當前專案）／ User-level（全機所有專案共用）。
  • 建法：可手寫，但推薦讓 Claude 生成——描述你要它做什麼，Claude 產出 name / description / system prompt。
  • 客製工具（想清楚它「真正需要」什麼）：類別 = Read-only / Edit / Execution / MCP / Other。
    例：code reviewer 不需 Edit（只讀+分析、不改），但可留 Execution 以辨識 pending changes。
  • 選 model：Haiku（快、輕）／ Sonnet（速度深度中庸）／ Opus（複雜分析）／ Inherit（跟主對話同一個）。
  • 選 color：UI 辨識用，多個 subagent 同時跑時好認。

6. Config 檔

存於 .claude/agents/your-agent-name.md。典型長相： 
---
name: code-quality-reviewer
description: Use this agent when you need to review recently written
  or modified code for quality, security, and best practice compliance.
tools: Bash, Glob, Grep, Read, WebFetch, WebSearch
model: sonnet
color: purple
---

You are an expert code reviewer specializing in quality assurance,
security best practices, and adherence to project standards. ...
欄位意義
name唯一識別；用 @agent code-quality-reviewer 引用。
description控制 Claude 何時用它。必須單行（要換行用 \n）。可放範例對話幫判斷。
tools可用工具清單（＝建立時選的，可隨時改）。
modelsonnet / opus / haiku / inherit。
colorUI 顏色。
Body（frontmatter 以下）＝ system prompt：給指令——focus 什麼、如何分析、如何回報 finding 給主 agent。好的 system prompt 是「有用 vs 沒抓到重點」的差別。

7. 讓 Claude 自動委派 + 測試

  • 想讓 Claude 不用你明講就委派 → description 裡放 "proactively"（例：Proactively suggest running this agent after major code changes…）。
  • 在 description 加具體範例對話，越具體 Claude 越知道何時委派。
  • 測試：改點 code，叫 Claude review。沒被用 → 回去把 description 的觸發情境/範例寫更具體。

8. 讓 subagent 真正有效的 4 個關鍵

爛配置的 subagent 會亂晃、跑太久、產出主 agent 沒法用的東西。四個修法：好描述 / 定義輸出格式 / 回報障礙 / 限制工具。
💡 config 資料怎麼被用：你發訊息給主 agent 時，每個可用 subagent 的 name + description 都被塞進主 agent 的 system prompt → 主 agent 據此決定何時 launch 哪個。而且主 agent launch 時要寫 input prompt，會拿 description 當寫 prompt 的指引。所以 description 不只控「何時跑」，還塑造「subagent 被告知做什麼」。
① 用 description 塑造 input prompt
  • code review subagent 泛泛描述 → 主 agent 寫出模糊的「use get diff to find current changes」（subagent 得自己猜哪些檔重要）。
  • 描述改成「You must tell the agent precisely which files you want it to review」→ 主 agent 就會寫出列出實際檔案的具體 input prompt。
  • 通用：web search subagent 描述加「return sources that can be cited」→ 委派時自動帶這指令。
② 定義輸出格式（單一最重要的改進）
  • (a) 製造自然停止點——每個 section 填完就知道做完了。
  • (b) 防跑太久——沒定義輸出的 subagent 難決定何時「研究夠了」，傾向遠超必要。
code review 範例輸出格式： 
1. Summary        4. Minor Issues
2. Critical Issues 5. Recommendations
3. Major Issues    6. Approval Status
填完六段 → subagent 知道可以停。
③ 回報障礙（Obstacles Encountered）
  • subagent 過程發現的 workaround（解了依賴、某指令要特定 flag）必須出現在回傳摘要，否則主線得重新發現一次，浪費時間 + token。
  • 要 surface：setup 問題/環境怪癖、發現的 workaround、需特殊 flag 的指令、造成問題的依賴/import。
  • 方法：在 output format 明確加一個 Obstacles Encountered section 才會可靠地被回報。
④ 限制工具：只給角色真正需要的工具 → 兩個好處：防非預期副作用 + 多個 subagent 時角色更清楚。
subagent 類型給的工具為什麼
Research / 唯讀Glob, Grep, Read不可能誤改檔案
Code reviewer+ Bash（跑 git diff）
但不給 Edit/Write		要看改了什麼，但不該改
Styling / 改碼+ Edit, Write它的工作就是真的改你的碼
🧩 小結：有效 subagent 的 4 特徵—— ① 具體 description（控何時 launch + 收什麼指令）② 結構化輸出（知道何時做完、回可用資訊）③ 障礙回報（主線不用重新發現）④ 限工具（唯讀給 research、bash 給 reviewer、edit/write 只給該改碼的）。四個各自簡單，合起來把 subagent 從「模糊地想幫忙」變成「準時完成、清楚回報的專注 worker」。

9. 何時該用 / 何時別用 subagent

🧭 唯一判準：中間過程對主線重不重要。exploration 與 execution 可分開 → 委派；每一步都依賴前一步的發現 → 留主線（別委派）。
✅ subagent 發光的三個甜區
  • Research（經典用法）：查陌生 codebase 的 auth 怎麼運作 → subagent 讀幾十個檔、追函式，全留自己 context，主線只收乾淨摘要（如「JWT 驗證在 middleware/auth.js:42，由 route/api.js 的 Express router 呼叫」）。
  • Code review：Claude 對「別人寫的碼」review 更犀利；同一 thread 造出來的碼、自己 review 會手軟（因為參與了創作、看不出問題）。reviewer subagent 在獨立 context 跑 git diff、套專屬 review 標準（可寫進 system prompt 讓全團隊一致）。
  • Custom system prompt：Claude Code 預設 prompt 偏「簡潔、程式導向」。copywriting subagent 給不同的 tone/audience/style（landing page、email 不要技術腔）；styling subagent 指向 design system 檔 → 執行時自動載入你的色彩變數/間距/元件慣例。
共通：要結果不要流水帳、探索會塞爆主 context、受益於 fresh perspective 或 custom prompt。
⛔ subagent 幫倒忙（3 個 anti-pattern）
  • Expert Claims：「you are a Python expert」「Kubernetes specialist」沒有價值——Claude 本來就有這知識，所謂專家 subagent 沒有主線做不到的事。
  • Sequential Pipelines：reproduce→debug→fix 三段 pipeline。只在任務真正獨立時可行；一旦每步依賴前一步的發現（debug 幾乎都是）→ handoff 時資訊流失。
  • Test Runners：回「tests failed」藏掉你診斷要的完整輸出，逼你再寫 debug script 去撈本來就有的細節。（實測中 test runner 是所有配置裡表現最差的。）
💡 launch subagent 的 overhead（失去可見性 + 把發現壓成摘要）只有在「subagent 做了主線做不到的事」時才划算。
🧭 決策規則（一句話記）
問自己：中間過程重要嗎？
· 不重要（只要最終結果）→ 委派 subagent。
· 重要（要邊看邊反應）→ 留主線。
✅ 用 subagent⛔ 別用 subagent
Research / 探索
Code review
需要 custom system prompt 的任務		沒有真本事的「專家」人設
每步依賴前一步的多階段 pipeline
跑測試（你需要完整輸出 debug）

🔟 澄清：三種 subagent 來源 + AGENTS.md ≠ .claude/agents

來源是什麼特點考點
靜態檔 .claude/agents/*.md課堂教的，預先定義可重用、description 自動委派、版控分享、/agents 管理D3
內建General / Explore / Plan開箱即用D1
動態產（Task / Workflow）runtime inline 生成，不寫檔彈性、一次性、隨任務組D1
⚠️ AGENTS.md ≠ .claude/agents/*.md：AGENTS.md 是「專案角色/指令文件」(像 CLAUDE.md，跨工具慣例)，不是 subagent 機制；.claude/agents/*.md 才是 Claude Code 能 spawn 的 subagent 定義。考卷問 subagent 時別把 AGENTS.md 拉進來答。
🔴 Tom 的盲點：一直靠動態產(Domain 1)+ 寫 AGENTS.md 選角，但從沒用過靜態 .claude/agents/*.md(Domain 3，20%)。補法：在任一專案打 /agents → Create 一個 code-reviewer → 打開產出的檔看 frontmatter(name / description[放 proactively 自動委派] / tools / model / color)。5 分鐘動手勝讀十遍。
🎯 扣 CCAR-F 考點：
  • Domain 1（1.2/1.3）：subagent context 隔離、不繼承 parent；Task 工具 spawn、allowedTools 要含 "Task"；context 要在 prompt 明確傳。← 本課「只回摘要、對話丟棄」正是這個。
  • Domain 3：.claude/agents/ 檔、scope（project vs user）、tools 限制、frontmatter 欄位。
  • Domain 5：用 subagent 做 context 隔離（Explore 隔離冗長探索、只回摘要保主 context）← 本課實例正是。
  • 選 model：Haiku/Sonnet/Opus/Inherit 對應「找→做→想」的成本考量（呼應你 CLAUDE.md 的 Agent Team 選模規則）。
