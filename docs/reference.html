CCAR-P 七大考域（配分 = job task analysis）

右側色塊 = Tom 落差自評：強 盤點已知 · 中 要補 · 缺 主攻。
17%

1. Solution Design & Architecture 強

  • 把商業問題轉成 Claude 方案
  • 端到端架構（input→processing→output→feedback loop）
  • 選架構樣式：workflow / agentic / augmented-LLM
  • 多 agent 系統與編排策略
  • 複雜問題分解（decomposition）
  • 對齊商業價值支柱（效率/轉型/生產力/成本/效能 SLA）
13%

2. Claude Models, Prompting & Context Eng 強

  • 依 trade-off 選 Claude 模型
  • system prompt / template / guardrail
  • prompt 技巧：zero-shot / few-shot / chain-of-thought
  • context window 最佳化與 token 管理
  • prompt 重用：caching、模組化、Skills
19%

3. Integration 中 最重

  • 評估 tool/agent 配置是否 capability bloat
  • 分析 authn/authz 找安全缺口
  • accuracy-latency trade-off 與設定決策
  • 大規模 observability 監控策略
  • 設計 RAG pipeline：chunking / indexing / retrieval（配對資料形狀與查詢型態）
  • 選連線協議：MCP / API·CLI / agent-to-agent
  • progressive discovery vs monolithic context
16%

4. Evaluation, Testing & Optimization 中

  • 定義 eval metrics（accuracy/latency/cost/safety/security）
  • 設計 eval dataset 與 mixed methodology 測試框架
  • A/B 測試與迭代
  • 診斷：prompt failure / hallucination / model mismatch
  • token/latency/cost 最佳化
  • 用 logging + observability 監控
14%

5. Governance, Safety & Risk 缺

  • guardrail 與 safety control
  • 辨識 LLM 系統風險/限制/failure mode
  • human-in-the-loop 驗證策略
  • 合規：GDPR / HIPAA / FedRAMP ← Tom 唯一紅色缺口，需背定義層級
  • 倫理：bias / fairness / transparency
14%

6. Stakeholder Communication & Lifecycle 中

  • structured discovery 與需求收集
  • 溝通架構決策與 trade-off
  • 管理利害關係人回饋與期望（含 SLA）
  • 架構文件與實作指引
  • lifecycle：discovery→design→handoff→monitor→iterate
7%

7. Developer Productivity & Operational Enablement 強

  • 為團隊配置 Claude 工具與環境（如 Claude Code）
  • 用 AI 輔助改善 dev workflow
  • 支援 debug 與維運問題排解

Tom 備考策略（集中火力）

7 域裡 6 域「強～中」，主攻只有 3 塊：
  • 缺 Domain 5 合規：GDPR / HIPAA / FedRAMP 各「管什麼、觸發條件、對 AI 系統要求」— 背定義即可，考選擇不是律師。
  • 中 Domain 4 正式 eval：eval dataset 設計、A/B、mixed methodology、hallucination 歸因。
  • 中 Domain 3 authz/observability 設計題：最小權限（移除多餘工具而非加 log）、大規模監控策略。
其餘（agent 架構 / Claude Code / prompt-context / MCP / token）= 盤點已知，對齊 Anthropic 官方用語即可。
