🎯 CCAR-F 攻略速記卡

Claude Certified Architect – Foundations · 5 域 · 60 題 / 120 分 / 720 過 / $125。情境題導向：官方 6 情境抽 4。重點在抓「錯誤選項(anti-pattern)」——這張證最愛考「哪個做法是錯的」。

💡 為什麼這張適合你：5 域全是技術核心（agent 架構 / MCP / Claude Code / prompt / context），沒有合規、沒有利害關係人、沒有 lifecycle——你日常在做的事。準備成本近乎零，主要是對齊官方用語與記住下面的 anti-pattern。

📋 6 大官方情境（抽 4 上場）

① 客服解決 agent：Agent SDK，處理退款/帳務/爭議，透過 MCP 工具(get_customer/lookup_order/process_refund/escalate_to_human)，目標 80%+ 首解率、知道何時升級。
② Claude Code 產碼：產碼/重構/debug/文件，客製 slash command、CLAUDE.md、plan mode vs 直接執行。
③ 多 agent 研究：coordinator 分派給搜尋/分析/綜合/報告 subagent，產出附引用報告。
④ 開發者生產力：探索陌生 codebase、理解 legacy、產樣板，用內建工具(Read/Write/Bash/Grep/Glob)+ MCP。
⑤ Claude Code CI/CD：自動 code review、產測試、PR 回饋，要可執行回饋、少 false positive。
⑥ 結構化資料擷取：從非結構化文件擷取、用 JSON schema 驗證、高準度、優雅處理 edge case。

Domain 1 · Agentic Architecture & Orchestration 27%

Agent loop 生命週期

  • 迴圈條件看 stop_reason："tool_use" → 繼續（執行工具、把結果 append 回對話）；"end_turn" → 終止。
⛔ 陷阱：用自然語言判斷是否該停 / 用「固定迭代上限」當主要停止機制 / 看助手有沒有輸出文字當完成訊號——這三個都是錯的，正解一律看 stop_reason。

多 agent：hub-and-spoke

  • coordinator 管所有 subagent 間通訊、錯誤處理、資訊路由。
  • subagent context 是隔離的，不會自動繼承 coordinator 的對話史 → 要的 context 必須在 prompt 裡明確傳。
  • Task 工具 spawn subagent；coordinator 的 allowedTools 必須含 "Task"。
  • 並行 subagent = 在同一個回應裡發多個 Task call（不是分幾個 turn）。
  • coordinator prompt 給「目標 + 品質標準」，不是逐步程序（讓 subagent 有適應力）。
⛔ 陷阱：coordinator 把任務切太細 → 廣泛研究題覆蓋不全。

確定性 vs 機率性：hooks & prerequisite gates

  • 需要「保證」的合規（如金流前先驗身分）→ 用 hooks / prerequisite gate，不能只靠 prompt（prompt 有非零失敗率）。
  • prerequisite gate：擋住 process_refund 直到 get_customer 回傳已驗證 ID。
  • PostToolUse hook：攔工具結果做正規化（Unix ts / ISO 8601 統一）；攔外送工具呼叫做合規（擋 >$500 退款→轉人工）。
✅ 心法：business rule 要「保證」→ hook；「盡量」→ prompt。

任務分解 & session

  • 可預測的多面向審查 → 固定 pipeline（prompt chaining，逐檔分析 + 一次跨檔整合 pass）。
  • 開放式調查（「幫 legacy 加完整測試」）→ 動態適應分解（先 map 結構→找高影響區→隨依賴發現調整計畫）。
  • --resume <session-name> 續特定對話；fork_session 從共同基線開平行分支。
⛔ 陷阱：拿舊 tool result 硬 resume。✅ 正解：工具結果過期時，開新 session + 灌結構化摘要比 resume 可靠。

Domain 2 · Tool Design & MCP Integration 18%

工具描述 = 選工具的主要依據

  • 描述太少/重疊 → 誤路由（analyze_content vs analyze_document 描述雷同）。
  • system prompt 的關鍵字會蓋過工具描述。
✅ 修法：改名消重疊（analyze_content→extract_web_results）；把通用工具拆成專用（analyze_document→extract_data_points/summarize_content/verify_claim_against_source）。

MCP 結構化錯誤

  • isError flag。錯誤分四類：transient(逾時) / validation(輸入錯) / business(違規) / permission。
  • 回傳 errorCategory + isRetryable boolean + 人類可讀說明。
  • 要分「access 失敗（需重試）」vs「valid empty result（查詢成功但無資料）」。
⛔ 陷阱：統一回「Operation failed」→ agent 無法做正確 recovery 決策。

工具數量 & tool_choice & MCP scoping

  • 工具太多（18 vs 4-5）→ 選擇可靠度下降。每個 subagent 只給角色需要的工具。
  • tool_choice："auto"(可回文字) / "any"(必呼叫某工具) / 強制 {"type":"tool","name":"..."}。
  • MCP scoping：專案級 .mcp.json（團隊共用）vs 使用者級 ~/.claude.json（個人）。用 ${GITHUB_TOKEN} 展開避免 commit 密鑰。
  • MCP resources 暴露內容目錄（issue 摘要/schema）減少探索性呼叫；標準整合優先用社群 MCP（Jira）。

內建工具

  • Grep=搜內容；Glob=檔名/副檔名 pattern（**/*.test.tsx）；Read/Write=整檔；Edit=唯一文字定位的精準改。
  • Edit 因文字不唯一失敗 → Read + Write fallback。
  • 漸進理解 codebase：先 Grep 找進入點 → Read 跟著 import 追流程，不要一次讀全部。

Domain 3 · Claude Code Configuration & Workflows 20%

CLAUDE.md 階層

  • 使用者級 ~/.claude/CLAUDE.md（不隨版控分享）→ 專案級 .claude/CLAUDE.md 或根目錄 → 目錄級（子資料夾）。
  • @import 引外部檔保持模組化；.claude/rules/ 放主題規則檔取代單一巨型 CLAUDE.md。
  • .claude/rules/ 檔的 YAML frontmatter paths（glob）→ 只在編輯符合檔案時載入，省 context/token。
⛔ 診斷題：新成員收不到指令 = 指令放在「使用者級」而非「專案級」。用 /memory 查哪些 memory 被載入。

Commands / Skills

  • 專案級 .claude/commands/（版控共享）vs 使用者級 ~/.claude/commands/。
  • Skills 在 .claude/skills/，SKILL.md frontmatter：context: fork（隔離 sub-agent 跑、輸出不污染主對話）、allowed-tools（限工具防破壞）、argument-hint。
  • Skill（按需、任務型）vs CLAUDE.md（always-load、通用標準）。

Plan mode vs 直接執行 & CI/CD

  • Plan mode：大規模改動、多方案、架構決策、多檔（如 library migration 影響 45+ 檔）。
  • 直接執行：簡單、範圍清楚（單檔 bugfix、加一個驗證）。可組合：plan 調查 + 直接執行實作。
  • CI/CD：-p(--print) 非互動模式防卡；--output-format json + --json-schema 出機器可解析結果貼 PR。
⛔ 陷阱：同一個「產碼的 session」去 review 自己的改動 → 較差。✅ 用獨立 review instance；重跑 review 時把前次 finding 灌進 context，只報新的/未解的，避免重複留言。

Domain 4 · Prompt Engineering & Structured Output 20%

明確 criteria > 模糊指令

  • 「只在註解與實際行為矛盾時 flag」>「檢查註解是否正確」。
  • 「be conservative」「只報高信心」無法提升精準度 → 要用具體分類 criteria。
  • 高 false-positive 分類會侵蝕開發者信任 → 暫時關掉爛分類、定義帶程式碼範例的嚴重度分級。

Few-shot

  • 詳細指令仍不一致時，few-shot 是最有效手段。給 2-4 個針對模糊情境、展示「為何選 A 而非合理替代」的範例。
  • few-shot 讓模型「泛化判斷」而非只 match 預設案例；也降擷取任務的幻覺。

結構化輸出 = tool use + JSON schema

  • tool_use + JSON schema = 保證 schema 合規的最可靠法，消 JSON 語法錯。
  • 但 schema 只防語法錯，不防語意錯（如 line item 加總不等於 total、值放錯欄）。
  • schema 設計：來源可能沒有的欄位設 optional/nullable，防模型為填必填欄而捏造。enum 加 "other"+detail 字串保擴充。
  • tool_choice: "any" 保證出結構化；強制 {"type":"tool","name":"extract_metadata"} 確保特定擷取先跑。

Domain 5 · Context Management & Reliability 15%

核心觀念

  • 長 context 最佳化、防 context drift/bloat（工具輸出 pruning、compaction）。
  • 用 subagent / 多步 workflow 做 context 隔離（如 Explore subagent 隔離冗長探索輸出、只回摘要，保住主對話 context）。
  • 錯誤傳播：subagent 內部先做 local recovery（transient 錯），只把「無法在本地解決的錯 + 部分結果 + 試過什麼」上報 coordinator。
  • token 經濟：prompt caching / cache check-pointing。
✅ 心法：多階段任務怕 context 爆 → 把冗長工作丟 subagent 做、只收摘要回來。
